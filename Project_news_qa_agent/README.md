# AI News Q&A Agent

관심 주제에 대한 최근 뉴스를 실시간으로 수집하고, 수집된 뉴스 범위 안에서만 근거를 들어 답변하는 Q&A 에이전트입니다. 세션 시작 시 한 번만 뉴스를 수집하며, 이후 질문은 별도의 크롤링 없이 인메모리 벡터스토어에서 처리합니다.

```
관심 분야를 입력하세요: 제논
검증 키워드를 입력하세요 (예: 전공 / 여러 개는 콤마로 구분): AI 기업
기간을 숫자로 입력하세요 (최근 며칠, 예: 7): 7
...
질문> 제논이 최근 집중하는 사업은 뭐야?
```

## 프로젝트 목표

일반적인 LLM은 학습 데이터 컷오프 이후의 이슈에 대해 답변하지 못하거나, 근거 없이 그럴듯한 답을 생성하는 경우가 있습니다. 이 프로젝트는 질문이 들어올 때마다 실시간으로 관련 뉴스를 먼저 수집하고, 그 안에서만 사실 근거를 찾도록 설계했습니다. 뉴스는 별도 DB에 적재하지 않고 세션 동안만 메모리에 유지합니다.

## 핵심 기능

- **주제 + 검증어 분리 입력**: 주제(topic)와 별도로 검증 키워드(validation terms)를 입력받아, 동음이의어나 문맥이 다른 기사를 걸러내는 기준을 사용자가 직접 통제할 수 있습니다. (예: 주제 "서울여대" + 검증어 "전공" → 학과/전공 관련 기사만 남김)
- **2단계 하이브리드 관련성 필터**: 임베딩 코사인 유사도로 명확한 사례를 빠르게 분류하고, 애매한 경계 구간만 LLM으로 정밀 재판정합니다. 전체 기사를 LLM으로 판정하는 방식보다 비용과 지연을 줄이면서도 의미 기반 판단을 유지합니다.
- **의미 기반 중복 제거**: 임베딩 유사도로 사실상 동일한 기사를 하나로 합칩니다. 필터링과 중복 제거는 같은 임베딩을 재사용하며, 중복 계산은 발생하지 않습니다.
- **근거 인라인 인용 답변**: 답변은 하나의 자연스러운 문단으로 작성되며, 수집된 기사로 뒷받침되는 문장에는 `(기사 N)` 형태로 출처를 표시합니다. 관련 뉴스가 없는 경우 일반 지식으로 답변을 완결한 뒤, 마지막에 그 사실을 짧게 밝힙니다.
- **필터 효과 정량 평가 도구**: 관련성 필터가 실제로 얼마나 효과적인지 Precision/Recall/F1/Accuracy로 측정할 수 있는 라벨링·평가 스크립트를 포함합니다.

## 동작 흐름

1. 관심 주제와 검증 키워드를 입력받습니다.
2. 수집 기간(일)을 입력받습니다.
3. LLM이 검색 키워드를 추출합니다. 고유명사는 그대로 유지하고, "AI 기업"처럼 지나치게 포괄적인 표현은 검색 키워드로 사용하지 않도록 프롬프트로 제한합니다. (검색 예산 낭비 방지)
4. 네이버 뉴스 검색 API로 메타데이터를 수집하고, 본문을 크롤링합니다.
5. 기사 본문을 임베딩한 뒤, 이를 재사용하여 관련성 필터(임베딩 1차 + LLM 경계 재판정)와 중복 제거를 수행합니다.
6. 정제된 기사로 FAISS 인메모리 벡터스토어를 구축합니다.
7. 질문 루프에 진입합니다. 질문마다 유사도 임계값을 넘는 기사만 재검색하여(개수 고정 아님) 답변을 생성합니다.

## 기술 스택

Python · LangChain(LCEL) · OpenAI(`gpt-4o-mini`, `text-embedding-3-small`) · FAISS(인메모리) · aiohttp/BeautifulSoup · 네이버 뉴스 검색 API(NCP)

## 폴더 구조

```
main.py                      # 엔트리포인트 (수집 세션 + 질문 루프)
build_labeled_set.py         # 평가용 골드셋 라벨링 도구
eval_filter.py                # 관련성 필터 효과 측정 (Precision/Recall/F1/Accuracy)
config.py                    # .env 로더
app/
  schemas.py                  # 데이터 모델
  query_understanding.py      # 주제 → 검색 키워드 추출
  news_search.py               # 네이버 뉴스 검색
  crawler.py                   # 본문 크롤링
  embeddings.py                 # 공유 임베딩 클라이언트
  relevance_filter.py          # 임베딩 + LLM 하이브리드 관련성 필터
  deduplication.py             # 의미 기반 중복 기사 제거
  vector_store.py              # FAISS 구축 + 유사도 기반 검색
  rag_chain.py                 # 답변 생성 (근거 인라인 인용)
```

## 시작하기

```bash
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
cp env.example.txt .env   # 발급받은 키 입력
python main.py
```

`.env`에 다음 세 가지가 필요합니다: `OPENAI_API_KEY`, `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET`.

네이버 API 키는 [console.ncloud.com](https://console.ncloud.com)에서 발급받습니다. 예전 개발자센터(`developers.naver.com`) 방식과 인증 헤더 구조가 다르므로 주의가 필요합니다. 현재 코드는 신규 방식(NCP API Gateway, `X-NCP-APIGW-*` 헤더)에 맞춰져 있습니다.

## 설정값 (config.py / .env)

| 변수 | 기본값 | 설명 |
|---|---|---|
| `RELEVANCE_LOW_THRESHOLD` | 0.15 | 이 값 미만이면 임베딩 1차 필터에서 명확히 무관한 기사로 즉시 제외 |
| `RELEVANCE_HIGH_THRESHOLD` | 0.35 | 이 값 이상이면 임베딩 1차 필터에서 명확히 관련 있는 기사로 즉시 통과 (두 값 사이는 LLM이 재판정) |
| `RELEVANCE_LLM_MODEL` | `gpt-4o-mini` | 경계 구간 기사를 재판정할 때 사용하는 모델 |
| `DEDUP_SIMILARITY_THRESHOLD` | 0.90 | 이 값 이상 유사하면 같은 기사로 간주 |
| `TOP_K_ARTICLES` | 5 | 답변 근거로 사용할 출처의 최대 개수 (상한, 고정 개수 아님) |
| `MAX_CRAWL_ARTICLES` | 20 | 검색 시 최대 수집 건수 |
| `CRAWL_TIMEOUT_SECONDS` | 8 | 크롤링 타임아웃 |

`RELEVANCE_LOW_THRESHOLD`, `RELEVANCE_HIGH_THRESHOLD`는 임베딩 모델과 도메인에 따라 최적값이 달라지므로, 아래 평가 도구로 실측 후 조정하는 것을 권장합니다.

## 관련성 필터 평가하기

관련성 필터가 실제로 효과가 있는지 정량적으로 검증할 수 있습니다.

```bash
# 1) 실제 기사를 수집하고 콘솔에서 y/n으로 관련성을 라벨링 (labeled_set.json에 누적 저장)
python build_labeled_set.py "제논" "AI기업" 20

# 2) 필터 적용 전(V0) / 적용 후(V1)의 Precision·Recall·F1·Accuracy 비교
python eval_filter.py
```

`eval_filter.py`는 `labeled_set.json`에 저장된 topic + 검증어 조합별로 자동으로 지표를 계산합니다.

### 측정 결과 예시

topic="제논", 검증어=["AI기업"] 조합으로 기사 16건(관련 8건 / 무관 8건)을 라벨링해 필터 적용 전(V0)과 하이브리드 필터 적용 후(V1)를 비교한 결과입니다.

| 지표 | V0 (필터 없음) | V1 (검증어 기반 하이브리드 필터) | 개선폭 |
|---|---|---|---|
| Precision | 0.500 | 0.727 | +0.227 |
| Recall | 1.000 | 1.000 | ±0.000 |
| F1 | 0.667 | 0.842 | +0.175 |
| Accuracy | 0.500 | 0.812 | +0.312 |

**해석**
- Recall이 1.0으로 유지됐다는 것은, 필터가 진짜 관련 있는 기사를 하나도 놓치지 않았다는 뜻입니다. 필터를 걸었을 때 흔히 우려되는 "과잉 필터링으로 좋은 기사까지 잘라내는" 부작용이 없었습니다.
- Precision이 0.5 → 0.727로 오른 것은, 필터 없이는 통과된 기사 절반이 무관한 기사였던 것이 필터 적용 후 73%가 실제 관련 기사로 개선됐다는 의미입니다. 원래 목표였던 "동음이의어/무관 기사 제거"가 실질적으로 작동하고 있음을 보여줍니다.
- 다만 이 결과는 표본 16건 기준으로, 방향성 확인에는 충분하지만 수치 자체의 신뢰구간은 넓습니다. 더 많은 topic·검증어 조합으로 라벨링을 누적할수록 결과가 안정화됩니다.

## 알려진 이슈 및 트러블슈팅

- **`httpx proxies` 에러**: `openai` 패키지와 최신 `httpx` 버전 간 충돌입니다. `httpx==0.27.2`로 고정해 해결했습니다.
- **네이버 API 401 오류**: 구 API(Client ID/Secret + `openapi.naver.com`)와 신규 NCP 방식의 인증 헤더가 다릅니다. 현재 코드는 신규 방식 기준입니다.
- **콘솔 한글 입력 깨짐**: PyCharm 콘솔과 한글 IME 조합 타이밍 문제로 발생합니다. UTF-8 강제 고정과 깨진 입력 감지 후 재입력을 구현했습니다. 문제가 지속되면 `python main.py "질문"`처럼 인자로 직접 전달하는 방법이 가장 안정적입니다.
- **검색어 과대 일반화**: LLM이 "AI 기업"처럼 지나치게 넓은 표현을 검색 키워드로 추출하면 검색 예산이 낭비됩니다. 고유명사는 유지하고 범용 표현은 검색 키워드로 사용하지 않도록 프롬프트에서 제한했습니다.
